难蚌实录二 | 使用 K3S 构建集群

单节点 Server 安装是一个功能齐全的 Kubernetes 集群，它包括了托管工作负载 pod 所需的所有数据存储、control plane、kubelet 和容器运行时组件——K3S 官方文档

K3S 主要分 Server 安装和 Agent 安装，粗糙理解就是主-从模式：Server 节点为主——负责部署集群、监测状态、任务调度等，Agent 节点为从——接收主节点指令执行计算任务等

树莓派 4B 外设相对少，但软件支持更好，Ubuntu 22.04 AArch64 Server 功能完整，适合用作 Server 节点

荔枝派 4A 外设丰富（甚至有 GPU 和 NPU），但软件支持较弱，操作系统和系统软件都需深度定制，勉强能跑 K3S（还有点小 BUG，详可参见“难蚌实录三”），适合用作 Agent 节点

安装 K3S 的具体操作：

  1. 在树莓派 4B 上 Server 安装
    1. 如果树莓派上有防火墙服务，则需要至少开放端口 6443 和 10250
    2. 参考 K3S-RiscV 即可
    3. 可以执行 systemctl enable k3s 允许 K3S 服务自启动，在 Agent 平台也有类似操作
  2. 在荔枝派 4A 上 Agent 安装
    1. 先安装 K3S 二进制程序，以及获取安装脚本 k3s-install.sh。依旧参考 K3S-RiscV即可
    2. 在 Server 上执行 cat /var/lib/rancher/k3s/server/node-token 获取加入集群需要的校验 token
    3. 在 Agent 平台执行 K3S-Agent 安装命令
      #/bin/bash
K3S_URL=https://<server-hostname | server-ip>:<port (default-6443)> \
K3S_TOKEN=<token> \
INSTALL_K3S_EXEC="agent" \
INSTALL_K3S_SKIP_DOWNLOAD="true" \
bash -x ./k3s-install.sh
      • 环境变量 K3S_URL=https://<server hostname | server ip>:<port (default-6443)> 使 Agent 可以找到 Server
        • 不论使用 server hostname 还是 server ip，都要确保 Agent 能通过该字段连接到 Server，可以执行 ping <server hostname | server ip> 测试连通性。不能保证设备可以获取固定静态 IP 的场景中建议使用 server hostname，不要写死 IP
        • 关于使用 mDNS 支持通过 hostname 解析到设备见“难蚌实录三”
      • 环境变量 K3S_TOKEN=<token> 使 Agent 可以加入 Server 开启的集群
      • 环境变量 INSTALL_K3S_EXEC="agent" 指定以 Agent 模式安装，该模式下 K3S 服务名称是 k3s-agent.service

难蚌实录三 | Hello 服务部署与 Pod 调度测试

使用 carvicsforth/hello-kubernetes 镜像（支持 ARM64 和 RISC-V 双架构），部署 2 个副本，K3s 会自动调度到两个节点——伟大的豆包

豆包给写的配置文件 hello-het.yaml

apiVersion: apps/v1
kind: Deployment
metadata:
  name: hello-het
spec:
  replicas: 2
  selector:
    matchLabels:
      app: hello-het
  template:
    metadata:
      labels:
        app: hello-het
    spec:
      containers:
      - name: hello-kubernetes
        image: carvicsforth/hello-kubernetes:1.10.1
        ports:
        - containerPort: 8080

在 Server 上执行 kubectl apply -f hello-het.yaml 即可部署，执行 kubectl delete -f hello-het.yaml 可定点清除部署的内容

难蚌时刻之 docker hub 被 BAN 难以直接拉取镜像

首先，K3S 自带基于 containerd 的容器运行时组件，安装 K3S 的平台无需额外安装 docker。另外， docker 和 containerd 都默认从 docker hub 拉镜像

有两个好用的国内代理站：dockerproxy.net，docker.xuanyuan.me，docker 和 containerd 的配置：

/* file name: daemon.json，不论是 apt 安装还是 snap 安装都是这个文件（路径小有不同），没有就创建 */
{
    "registry-mirrors": [
        "https://dockerproxy.net/",
        "https://docker.xuanyuan.me/"
    ],
}

# file name: /etc/containerd/config.toml，也是没有就创建
[plugins."io.containerd.grpc.v1.cri".registry]
    [plugins."io.containerd.grpc.v1.cri".registry.mirrors]
        [plugins."io.containerd.grpc.v1.cri".registry.mirrors."docker.io"]
            endpoint = ["https://dockerproxy.net", "https://docker.xuanyuan.me"]
难蚌时刻之 LicheePi 4A 作为 Agent 加入 cluster 后不能自动生成网络配置文件

基于上面的 hello-het.yaml 进行部署后，执行 kubectl get pod -o wide 查看 pod 创建运行情况，出事情了（意味深）：

在 lpi4a 节点上的 pod 处于非 READY 状态，看得人浑身刺挠

可以执行 kubectl describe pod pod <pod-name> 查看部署该 pod 时的日志。本例中报告的错误是：

Failed to create pod sandbox: rpc error: \
code = Unknown desc = failed to setup network for sandbox "76...": \
plugin type="flannel" failed (add): \
open /run/flannel/subnet.env: no such file or directory

我查了一天 AI（的确是一天🌚，也的确是 AI🌚），都在说 /run/flannel/subnet.env 应该是 k3s-agent 安装启动并加入集群时自动从 Server 节点上扒到本地的，如果没有那就是安装或者启动时出了问题建议重装或重启，我尝试了重装重启，没效果

其实直到现在我还是不知道为什么（

豆包大人说，可能是 RiscV 的 LicheePi 4A 对 K3S 的移植不完善，勉强能运行 K3S 就已经很不错了（意味深），无力创建 subnet.env 属实可以理解

是啊，理解万岁（

那么最简单且目前看来唯一有效的办法就是在 Agent 上手工制作一份 subnet.env，可以手写，也可以用 scp 什么的直接从 Server 上获取。有了 subnet.env，再重启一下 k3s-agent，刚才失败的部署会自动重试，然后就 OK 了

值得注意的是，/run 通常是由 tmpfs 挂载的，这是一个内存中的临时文件系统，每次系统重启挂载都会失效，/run 会被清空。对于可以自动创建 /run/flannel/subnet.v 的节点来说这无所谓，但对于本例中的 LicheePi 4A，如果没有更好的方法一劳永逸地解决问题，那么每次重启都要手动重复一遍上面的操作，哎哟我

下面的难蚌时刻都是为了解决这个问题附带产生的

难蚌时刻之 Server 节点和 Agent 节点需要同步时间

所有节点的系统时间需同步（timedatectl），证书/Token 过期也会导致通信失败——伟大的 DeepSeek

DeepSeek 大人说的有道理啊，于是在两块板子上都做了一下时间同步，同步到国内的一些 NTP 服务器，具体操作可以询问伟大的 DeepSeek。Anyway，时间同步没有解决上面的问题

难蚌时刻之使用 mDNS 支持外部设备从 hostname 解析主机 IP

这个操作产生于对设备连接热点网络后 IP 动态分配无法固定的考虑。具体地，在 Server 和 Agent 上都安装并启动 avahi-daemon 即可

最难蚌时刻之在 Ubuntu 22.04 On RaspberryPi4 使用 netplan 配置静态 IP 搞崩热点连接导致 ssh 会话关闭进而失去对开发板的控制

如果每次启动后本机 IP 都不一样（开了 DHCP 动态 IP 申请），k3s server 配置大概率会崩，所以得想办法搞出静态 IP

ubuntu 官方发行版使用 netplan 工具而不是 dhcpcd.conf 文件配置本机 IP，具体配置方法可自行查询网络。在改动配置之前，一定要确保除了 ssh 无线连接 IPV4 地址之外，还有其它可靠手段连接板子！！！！！

还好树莓派相关的 server 类 ubuntu 发行版都配置了串口连接：树莓派4B——串口通信配置与使用指南
